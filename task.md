
请编写一个使用 Python 实现的桌面 GUI 工具，用于从 llspace 平台导出用户卡片内容。开发需严格遵循以下要求：

### 环境与依赖
- 使用 `uv` 创建项目环境。
- GUI 使用标准库 `tkinter` 实现。
- 网络请求使用 `requests`。
- 所有 API 调用行为（包括端点路径、请求方法、请求头、请求体结构、认证方式、响应字段）**必须严格依据项目根目录下的 `API_specs.md` 文档**。**不得自行推测或解释 API 行为**，若文档未明确定义某接口细节，则应跳过该功能或报错提示“API 规范缺失”。

### 核心流程
1. **登录界面**  
   - 启动后首先显示输入框，收集用户的 llspace **用户名** 和 **密码**。
   - 点击“登录”后，依据 `API_specs.md` 调用用户登录接口，获取有效认证凭据（如 token）。

2. **加载并展示卡包**  
   - 登录成功后，依据 `API_specs.md` 调用卡包列表接口，获取用户所有卡包。
   - 在 GUI 主界面显示：
     - 用户信息（仅展示 `API_specs.md` 中明确返回的字段，如 `name`、`avatar`）
     - 卡包列表（仅使用 `API_specs.md` 中定义的字段，如 `pg_name`、`cover_url` 等）
   - 所有获取到的原始响应数据需缓存为本地 JSON（如 `./cache/session_data.json`）。

3. **导出选中卡包**  
   - 用户选择一个卡包后，点击“导出”：
     - 依据 `API_specs.md`，调用卡包目录接口（`/api/1/pg/directoryList` 或文档指定路径）获取卡片 ID 列表。
     - 对每张卡片，依据 `API_specs.md` 调用卡片详情接口（`/api/1/cards/detail` 或文档指定路径），传入文档要求的参数（如 `card_id`、`pg_id` 等）。
     - 仅使用 `API_specs.md` 中明确定义的字段（如 `title`、`created_date`、`description`、`cover_url`、`url`）进行后续处理。

4. **本地保存内容**  
   - 每个卡包导出到独立目录，命名格式：`{safe_pg_name}_{timestamp}`。
   - **Markdown 文件**：
     - 保存路径：`{卡包目录}/{卡包名}.md`
     - 内容包含每张卡片的 `title`、格式化后的 `created_date`、`description`、本地化的 `cover_url` 图片引用。
     - 卡片按 `created_date` 升序排列。
   - **图片资源**：
     - 所有 `cover_url` 下载至 `{卡包目录}/images/`，使用安全文件名。
   - **网页快照**（若 `url` 字段存在）：
     - 下载完整 HTML 页面至 `{卡包目录}/web/{card_id}.html`
     - 页面内所有资源（图片、CSS、JS）需重写为本地相对路径，并保存到 `web/` 子目录，确保离线可正常浏览。
     - 生成 `{卡包目录}/index.html`，列出所有卡片快照链接。

5. **进度反馈**  
   - GUI 实时显示：
     - 当前卡包名称
     - 当前处理的卡片（显示 `title` 或 ID）
     - 已完成卡片数 / 总卡片数
     - 总体进度条（百分比）
   - 所有网络与文件操作必须在后台线程执行，不得阻塞 GUI。

6. **导出完成**  
   - 显示摘要：导出卡包数、卡片总数、导出根路径。
   - 提供“打开文件夹”按钮（调用系统默认文件浏览器）。

7. **健壮性**  
   - 对 API 错误、网络超时、无效响应等，记录日志（`export.log`），跳过失败项并继续。
   - 若 `API_specs.md` 中某接口字段缺失或格式不明，**不得猜测字段含义**，应记录警告并跳过相关功能。

### 交付物
- 单文件 Python 脚本 `llspace_exporter.py`
- 代码结构清晰，含必要注释
- 遵循 PEP 8
- GUI 布局简洁、响应良好
